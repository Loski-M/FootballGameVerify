# Python 验证比赛模拟系统架构与实现思路

## 文档目标

这份文档的目标不是设计正式游戏工程，而是定义一个**独立于 Godot 的 Python 验证程序**，用于尽快回答下面这些问题：

1. 球员 AI 的基础建模是否成立。
2. 球权状态机是否能驱动出像足球的比赛过程。
3. 持球决策、无球接应、施压回收这些规则是否能形成稳定节奏。
4. 参数是否可以通过批量模拟和日志分析进行调优。
5. 后续哪些逻辑可以较平滑地迁回 Godot。

---

## 这套 Python 程序的定位

它不是：

- 正式游戏客户端
- 最终可玩的版本
- 完整视觉化模拟器

它是：

- **比赛模拟内核验证器**
- **AI 规则实验场**
- **参数调优工具**

因此它优先强调：

1. 逻辑清晰
2. 批量可跑
3. 日志可读
4. 容易重构

而不是：

1. 动画表现
2. 精美界面
3. 操作手感

---

## 本阶段最小目标

第一版 Python 验证程序只需要做到这些：

1. 能跑完一场简化 5v5 比赛。
2. 有固定步长的比赛推进。
3. 有队伍级状态和球员级动作。
4. 有最基础的持球动作：带、传、射。
5. 有最基础的无球动作：接应、拉开、施压、回收。
6. 能输出比赛事件日志和统计摘要。

第一版**不追求**：

- 精细碰撞
- 复杂犯规
- 定位球体系
- 能量和大招
- 玩家实时前压/退守
- 图形界面

这些都可以在后续版本逐步加上。

---

## 为什么先做文本化模拟

这一步最需要验证的是：

- 规则是否成立
- AI 是否合理
- 参数是否平衡

文本化模拟的优势是：

1. 一次能跑很多场。
2. 更容易看统计分布。
3. 改规则之后反馈很快。
4. 能逼着我们把逻辑层和表现层彻底分开。

后续如果需要，可以再给 Python 版加一个非常轻量的 2D 调试可视化，但不应该作为第一优先级。

---

## 总体架构

推荐把 Python 验证版拆成下面几层：

1. **数据层**
2. **比赛状态层**
3. **AI 决策层**
4. **模拟推进层**
5. **日志与统计层**
6. **程序入口层**

---

## 1. 数据层

负责描述“稳定不变或低频变化的配置”。

建议包含：

- 球员基础属性
- 球员派生能力
- 球队风格参数
- 球场尺寸
- 模拟频率
- 动作阈值

### 设计原则

1. 基础属性和派生能力分开。
2. 尽量使用 dataclass。
3. 这一层不放 AI 逻辑。

### 典型对象

- `PlayerAttributes`
- `PlayerDerived`
- `TeamTactics`
- `MatchConfig`

---

## 2. 比赛状态层

负责描述“比赛此刻发生了什么”。

建议包含：

- 球员当前位置
- 球员速度
- 球员体能
- 当前目标点
- 当前动作
- 当前是否持球
- 球的位置与速度
- 当前控球队
- 当前队伍状态
- 比赛时间

### 典型对象

- `PlayerState`
- `BallState`
- `TeamState`
- `MatchState`

### 设计原则

1. 这一层只保存状态，不直接做复杂判断。
2. 尽量避免把“规则”和“数据”混在同一个类里。

---

## 3. AI 决策层

这是 Python 验证版最重要的部分。

推荐采用：

- **队伍层：状态机**
- **球员层：评分式决策**

### 队伍层状态机

第一版建议只保留少数状态：

- `OrganizedPossession`
- `TransitionAttack`
- `DefensiveBlock`
- `HighPress`

这些状态用于控制：

- 队伍整体站位深度
- 接应积极性
- 压迫积极性
- 回收优先级

### 球员层动作

持球球员：

- `dribble`
- `pass`
- `shoot`

无球球员：

- `support`
- `spread`
- `press`
- `recover`

### 决策方法

持球球员对候选动作打分：

`总分 = 基础倾向 + 属性修正 + 场景修正 + 风格修正 + 微随机`

示例：

- `dribble` 看前方空间、最近防守者距离、盘带能力
- `pass` 看接应点质量、传球线路长度、传球能力
- `shoot` 看距离、角度、射门能力、压迫程度

无球球员不需要在第一版做复杂全局搜索，只需要根据自身角色和当前比赛状态去生成目标点。

---

## 4. 模拟推进层

这一层负责按固定步长推动比赛。

第一版建议：

- `dt = 0.2`
- 决策更新频率：`0.4 ~ 0.6 秒`

这样比最终实时游戏更粗一些，但足够做第一轮逻辑验证。

### 每个 tick 的基本流程

1. 更新比赛上下文。
2. 更新队伍状态。
3. 更新需要重算的球员决策。
4. 执行动作。
5. 更新球员位置和球位置。
6. 处理传球接球、射门、扑救、球权变化。
7. 记录事件。

### 为什么第一版 dt 可以更大

因为 Python 验证版当前目标不是“视觉连续性”，而是：

- 规则验证
- 事件节奏
- 统计分布

等逻辑稳定以后，再把 dt 变小即可。

---

## 5. 日志与统计层

这是 Python 验证版的价值核心之一。

程序必须能输出：

- 进球
- 射门
- 传球成功
- 传球失误
- 抢断成功
- 回合开始
- 回合结束

同时还要统计：

- 最终比分
- 每队射门数
- 每队射正数
- 每队传球成功率
- 每队抢断数
- 每队控球时间

### 设计原则

1. 日志要尽量简洁、可读。
2. 统计要与日志解耦。
3. 未来便于导出 CSV 或 JSON。

---

## 6. 程序入口层

入口层负责：

1. 生成测试球员与球队。
2. 创建比赛配置。
3. 运行一场比赛。
4. 打印日志与统计摘要。

第一版建议提供两个入口：

- `run_match.py`：跑一场并输出详细日志
- `run_batch.py`：未来扩展为批量跑多场

本次先实现单场入口即可。

---

## 文件结构建议

第一版推荐结构：

```text
python_sim/
  __init__.py
  config.py
  models.py
  ai.py
  simulation.py
  sample_data.py
  reporting.py
run_match.py
```

### 各文件职责

- `config.py`
  放比赛配置、球场参数、动作阈值。

- `models.py`
  放 dataclass 和枚举。

- `ai.py`
  放队伍状态机、持球决策、无球目标点逻辑。

- `simulation.py`
  放比赛主循环和事件结算。

- `sample_data.py`
  放测试球队与测试球员生成器。

- `reporting.py`
  放日志格式化和统计输出。

- `run_match.py`
  程序入口。

## 第一版 AI 原则

### 持球 AI

只回答一个问题：

“现在我该带、传还是射？”

### 无球 AI

只回答两个问题：

1. 我该站到哪里？
2. 我现在更应该接应还是施压？

### 队伍 AI

只回答一个问题：

“我们当前是稳控、快推、回收还是高压？”

这能把复杂度控制在一个可实现范围内。

---

## 第一版暂时不做的内容

为了保证程序能尽快落地，下面这些先不做：

- 玩家输入层
- `pressureBias`
- 能量系统
- 大招
- 犯规和黄牌
- 定位球细节
- 复杂门将出击
- 复杂挡拆和战术套路
- 复杂碰撞和卡位

原因不是它们不重要，而是这些都建立在“基础比赛结构已经成立”之上。

---

## 从 Python 迁回 Godot 时，哪些东西可以复用

最值得复用的是：

1. 数据结构思想
2. 队伍状态机
3. 动作评分公式
4. 无球目标点规则
5. 事件日志与统计口径

不需要直接一比一照抄的，是：

1. Python 的程序组织方式
2. 具体数学实现细节
3. 过于粗糙的步长设置

Godot 版要承接的是“规则已经被证明有效”，而不是逐行翻译代码。

---

## 实施顺序

本轮实现建议按这个顺序落地：

1. 先搭数据结构和配置。
2. 再搭比赛状态与样例数据。
3. 再写最小 AI 决策。
4. 再写比赛主循环。
5. 再补日志和统计。
6. 最后跑一场测试比赛。

---

## 本轮实现成功标准

只要达到下面几个点，就算第一版成功：

1. 程序能稳定跑完一场。
2. 日志里能看到传球、射门、进球、断球。
3. 双方球员不会长期全部挤成一团。
4. 至少能看出一点控球和防守节奏差异。
5. 输出的统计结果基本合理。

达到这个程度后，就已经可以开始进入：

- 调参
- 批量跑
- 增加前压/退守
- 增加能量/大招
