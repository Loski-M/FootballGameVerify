## 二、最终实现概览

### 2.1 高空球被建模为新的“飞行维度”，不是新的 `PassType`

本轮没有把系统改成：

- `TO_FEET`
- `LEAD_PASS`
- `THROUGH_PASS`
- `LOFTED_PASS`

而是保留原有传球意图：

- `TO_FEET`
- `LEAD_PASS`
- `THROUGH_PASS`

同时新增：

```python
BallFlightType: GROUND | LOFTED
```

这样高空球与地面球成为“球如何飞”的维度，而不是“球为什么传”的维度。

### 2.2 数据结构已经扩成 2.5D

`models.py` 本轮新增或扩展了：

- `BallFlightType`
- `Intent.flight_type`
- `Intent.landing_x / landing_y`
- `Intent.vertical_speed`
- `BallState.z / vz`
- `BallState.flight_type`
- `BallState.landing_x / landing_y`
- `BallSnapshot.z`
- `BallSnapshot.flight_type`
- `BallSnapshot.landing_x / landing_y`

这意味着：

- AI 可以提前知道这是地面球还是高空球
- 模拟器可以完整结算空中飞行与落地
- 可视化和回放可以直接读取球高和落点

### 2.3 共享传球预览层已升级为“双球路评估”

`pass_logic.py` 现在的 `PassPreview` 已扩展为同时描述：

- `flight_type`
- `landing_x / landing_y`
- `vertical_speed`
- `time_to_landing`
- `peak_height`

`preview_pass_option()` 现在的结构是：

1. 先生成一份 **ground preview**
2. 再判断能否生成 **lofted preview**
3. 最后在两者之间选风险更低、场景更合理的方案

这一步让高空球完全接入了现有的共享传球规划层，而不是另开一套旁路逻辑。

### 2.4 第一版高空球物理已经落地

当前高空球的物理分两段：

#### 空中阶段

```python
z += vz * dt
vz -= gravity * dt
```

#### 落地阶段

当 `z <= 0` 且正在下落时：

- 球落地
- `z = 0`
- `vz = 0`
- 平面速度乘上 `lofted_pass_landing_speed_factor`
- `flight_type` 切回 `GROUND`

这个实现仍然是 top-down 2D 模拟，但已经足够支持：

- 球起高
- 球下落
- 落地后继续滚动

### 2.5 高空球与地面球的风险结构已经分开

当前地面球仍然主要看：

- 静态线路封堵
- 中途动态拦截
- 终点压力

而高空球则改成：

- 降低中前段地面封线影响
- 更强调落点压力
- 更强调“谁先到落点”

换句话说，高空球现在的思路是：

- 不是“穿过去”
- 而是“越过去后争第二落点”

### 2.6 控球判定已加入高度约束

新增规则：

- 外场球员只能在 `z <= ball_control_max_height_outfield` 时直接控球
- 门将可在 `z <= ball_control_max_height_goalkeeper` 时控高球

当前默认值：

```python
ball_control_max_height_outfield = 1.2
ball_control_max_height_goalkeeper = 2.4
```

这样避免了高空球在半空中被原有 2D 控球逻辑不合理“吸住”。

### 2.7 接球人与防守人已开始围绕落点跑位

`ai.py::choose_intercept_point()` 现在面对高空球时，会先看：

- 这名球员对这个高度是否可控
- 球何时下降到可控高度
- 自己能否赶到预计落点

如果可行，就直接把：

- `intercept_x`
- `intercept_y`

锁定到 `landing_x / landing_y`，也就是高空球的落点。

这让高空球开始具备“提前争落点”的行为，而不是沿用地面球那种只追当前球坐标的方式。

### 2.8 可视化已支持高空球调试

`visualizer.py` 本轮增加了：

1. **球影**
   - 地面位置处的深色圆点
2. **空中球体**
   - 随 `z` 向上偏移显示
   - 高空球会带白色描边
3. **落点环**
   - 高空球时显示 `landing_x / landing_y`
4. **侧边信息面板**
   - `ball height`
   - `flight type`
   - `landing`

这样即使不看代码，也能直接在回放里观察：

- 球有没有真正飞起来
- 会落在哪里
- 接球人是不是在争正确的落点

### 2.9 日志已经能区分地面球和高空球

传球日志现已输出为：

```text
(lead_pass)/ground
(lead_pass)/lofted
```

因此即使不开回放，也能从比赛报告直接看到高空球是否被实际使用。

## 四、实现过程中的实际取舍

### 4.1 高空球目前只接入了“传球”，没有接入“射门”

本轮高空球只用于：

- `PASS`

没有把：

- 射门高度
- 吊射
- 头球攻门

一起做进去。  
这是有意控制复杂度的结果，避免第一版同时把球路系统和射门系统一起打散。

### 4.2 没有做完整争顶对抗

当前高空球的落点争夺仍然是简化版：

- 先跑到落点
- 等球下降到可控高度
- 再由现有控球规则决定谁拿到球

还没有加入：

- 头球能力
- 身体对抗
- 争顶优先级

但作为验证版，这已经足够判断“第二落点机制”是否成立。

### 4.3 可视化更偏调试，而不是表演

本轮可视化没有追求华丽动画，而是优先让这几件事变清楚：

- 球在不在空中
- 球有多高
- 球会落在哪

这更符合当前 Python 验证版的定位。

## 五、本轮参数微调结果

高空球第一版跑通后，又做了一轮偏保守的参数调整，目标是：

- 减少边路球员随手起高球
- 让高空球更多集中到 GK、ANCHOR、长距离转移和明显绕封线场景

最终调整后的关键参数为：

```python
lofted_pass_min_distance = 12.5
lofted_pass_lane_relief = 0.55
lofted_pass_terminal_pressure_weight = 1.8
lofted_pass_receiver_space_weight = 0.9
lofted_pass_decision_bias = 1.45
lofted_pass_gk_decision_bias = 0.7
```

同时，高空球候选逻辑也收紧为：

- GK：始终可考虑高空球
- ANCHOR：满足距离阈值后可考虑
- 其他外场球员：必须更长距离，且更像真正的前送或转移场景
