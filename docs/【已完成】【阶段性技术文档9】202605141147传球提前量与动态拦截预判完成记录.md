## 二、最终实现概览

### 2.1 新增共享传球预览层

新增文件：

- `python_sim/pass_logic.py`

新增核心结构：

- `PassPreview`
- `preview_pass_option(match, team, passer, receiver, config)`

`PassPreview` 统一产出：

- `pass_type`
- `target_x`, `target_y`
- `ball_speed`
- `flight_time`
- `desired_lead`
- `run_dir_x`, `run_dir_y`
- `lane_static_penalty`
- `lane_dynamic_penalty`
- `terminal_pressure`
- `is_blocked`

这一步把原来分散在 `ai.py` 和 `simulation.py` 里的传球落点、球速和风险判断集中到了同一层。

### 2.2 采用了“方案 B”的缓存思路，但实现比原计划更轻

原先讨论的方案 B 是：

- 把预览结果写进 `Intent`
- 让执行阶段直接复用

这次代码最终确实走了这个方向，但没有给 `models.py` 额外增加新缓存字段，而是直接复用了 `Intent` 现有字段：

- `target_x`
- `target_y`
- `target_player_id`
- `pass_type`
- `pass_speed`

也就是说，AI 在生成 `PASS` 意图时，写进去的已经不是“接球人当前位置”，而是**预览后的真实落点和真实球速**。  
执行阶段会优先使用这份缓存，因此 AI 与执行不再各算各的。

### 2.3 提前量方向改成“意图优先”

在 `pass_logic.py` 中，接球跑动方向的推断优先级改为：

1. `intercept` 锁定方向
2. `intent.target_x/target_y` 方向
3. 当前 `vx/vy`
4. 球队进攻方向

这样可以减少“因为接球人当前瞬时速度不稳定，导致提前量推到身后”的问题。

### 2.4 加入动态拦截预判

当前实现不再只看静态传球线。

新增参数：

```python
pass_intercept_sample_dt: float = 0.1
pass_intercept_time_margin: float = 0.15
pass_intercept_hard_block_margin: float = 0.05
pass_dynamic_lane_weight: float = 1.3
pass_terminal_pressure_weight: float = 0.55
pass_dribble_safety_margin: float = 0.35
```

动态拦截逻辑为：

1. 沿球路离散采样
2. 对每个对手计算其跑到采样点的预计时间
3. 与球到该点的时间比较
4. 若能明显抢到中前段采样点，则直接标记 `is_blocked=True`
5. 若只是具备一般干扰能力，则转化为软惩罚

这样候选传球会先经过一轮“能不能传”的筛选，再进入收益比较。

### 2.5 接入“无安全传球则带球”

在 `ai.py::decide_ball_owner()` 中，live-ball 下：

1. 先计算 `shoot_score`
2. 再计算 `dribble_score`
3. 再获取 `score_best_pass()` 的最佳安全传球
4. 如果最佳传球分低于 `dribble_score + pass_dribble_safety_margin`
5. 则让 `PASS` 退出竞争

这样当线路明显不安全，或者虽然能传但收益不足时，球员会更自然地回退到带球推进。

### 2.6 restart 规则保持不变

死球重开依然保持：

- 开球
- 边线球
- 角球
- 球门球

第一动作强制为 `PASS`。

但其传球目标现在同样会先经过共享预览层；若最佳安全传球为空，则回退到最近队友并生成一份受预览逻辑约束的传球计划。

### 2.7 执行阶段改为直接复用预览结果

`simulation.py::_resolve_pass_target()` 现在优先读取：

- `owner.state.intent.target_x`
- `owner.state.intent.target_y`
- `owner.state.intent.pass_speed`

若缓存缺失，才退回调用 `preview_pass_option()` 重新生成。

这一步真正消除了之前的核心结构问题：

- AI 评估一条线路
- 执行层却踢另一条线路

### 2.8 `lead_defender_nudge_strength` 已真正接入

此前这个参数虽然存在于 `config.py`，但没有真正参与终点压力评估。  
本轮它已经接入 `pass_logic.py::_evaluate_terminal_pressure()`，用于控制高压终点的惩罚强度。

### 2.9 移除了旧的重复主流程逻辑

由于传球类型判断和球速判断已经收敛到共享预览层，本轮从 `ai.py` 中移除了旧版主流程使用的：

- `choose_pass_type()`
- `choose_pass_speed()`

避免后续维护时误以为系统里还存在两套并行的传球主逻辑。

## 四、和原计划相比的实际取舍

本轮与最初方案相比，有两个实际取舍：

### 4.1 没有扩展 `models.py`

原计划里的“方案 B”是可能新增：

- `planned_target_x`
- `planned_target_y`
- `planned_ball_speed`

最终没有这么做。原因是当前 `Intent` 现有字段已经足够承载同一份预览结果，继续加字段会让数据结构变重，但收益不大。

### 4.2 可视化增强没有在这一轮并入

原先的方案里提到：

- 预览落点
- 被阻断的传球线
- 动态拦截候选点

这部分本轮没有一起做。  
本轮只先完成行为逻辑闭环，避免把“逻辑正确性验证”和“可视化表现增强”混在同一次修改里增加排查难度。

## 七、结论

第九轮的目标已经完成，而且完成方式比最初方案更收敛：

- 不是继续叠参数

- **先预览球怎么飞**
